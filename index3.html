C-0

最終回答は10秒。でも実際は1分32秒

Reasoning型AIの投資判断では、画面に表示された回答だけで処理量を測ってはいけない。たとえばVisible Outputが500トークン、内部Reasoningが4,096トークン、Decode速度が50 tok/sなら、画面上の回答だけを生成する時間は10秒だが、実際の総生成量は4,596トークンとなり、完了まで約1分32秒かかる。

9.19×実際の生成量は、見える回答の約9.19倍（この仮定での比較例）

この例では、実際の生成量は見える回答の9.19倍であり、Qwen3-32B相当のBF16 KVではReasoningだけで約1GiBの追加メモリを使う。これは特定の仮定による比較例で、一般的な平均Reasoning量ではない。しかし、「短い回答だから処理も軽い」と判断すると、待ち時間、GPU容量、同時実行数を大きく見誤ることを示している。

C-1

AIサービスには5つの予算がある

AIサービスの基盤計画は、単一の「GPU性能」ではなく、五つの予算として整理すると判断しやすい。第一はモデルと作業領域が載るかを決める容量予算、第二はトークンをどれだけ速く生成できるかを決める帯域と速度の予算、第三は一度にどれだけの情報を渡すかというコンテキスト予算である。

第四は、どの仕事へどれだけReasoningを使わせるかという思考予算である。第五は、会話や案件情報をどこへ保存し、誰が更新・削除・監査するかという長期記憶の予算である。大きなモデルを購入しても、この五つが自動的に最適化されるわけではない。ワークロードごとに必要な容量、速度、Context、Effort、Memoryを割り当てることが投資配分の中心となる。

経営上の質問技術指標利用者への影響
載るか？（Fit）VRAM 容量そもそも使えるか
速いか？（Speed）User tok/s・TTFV・帯域待ち時間
どれだけ渡せるか？（Context）Application context精度・コスト
どれだけ考えさせるか？（Think）Reasoning Effort品質と待ち時間の両方
覚えられるか？（Remember）Memory Service継続性・統制
C-2

容量が大きい機械と、速い機械は違う

原理 ｜ メモリ律速GPUを宣伝上のFLOPSだけで選ぶと、LLM推論の投資判断を誤るおそれがあります。
  • decodeは演算性能より、メモリ帯域が律速になりやすいです。
  • 性能は、重みとKVを速く少ない量で読めるかで決まります。
  • 投資は帯域確保と、重み・KVの読出量削減にも向けます。

参考: Qiita「なぜLLMは遅いのか？ KV CacheとSpeculative Decodingの数学的理解」

ハードウェア選定では、メモリ容量をモデル搭載可否の指標、メモリ帯域を主にBatch 1 Decode速度の指標として分ける必要がある。RTX 5090は32GBと容量が小さい一方、理論帯域は1,792GB/sである。DGX Sparkは128GBを持つが273GB/sであり、同じ「AI向けハードウェア」でも得意なワークロードが異なる。

短いコンテキストのDense Q4モデルという限定条件では、RTX 5090上の32Bモデルは理論112 tok/sだが、70Bの理論重み35GBは搭載できない。DGX Sparkは70Bを載せられる容量がある一方、帯域Rooflineは7.8 tok/sである。Mac Studio M3 Ultraは最大512GBと819GB/sを持ち、大規模モデルを単一の統合メモリへ置きやすいが、A100 80GB SXMの2,039GB/sより帯域は低い。

ハードウェア容量帯域32B Q4 理論tok/s70B搭載
RTX 509032 GB1,792 GB/s112×
RTX 309024 GB936 GB/s58.5×
A100 80GB SXM80 GB2,039 GB/s127○
DGX Spark128 GB273 GB/s17.1○ (7.8 tok/s)

したがって、製品を単純な順位で並べるのではなく、「対象モデルが載るか」「一人当たり何tok/s必要か」「何人が同時に使うか」「長いContextを使うか」で選ぶべきである。下表のtok/sはKernelや通信を無視した理論Rooflineであり、購入判断には同じRuntimeとワークロードによる実測が必要である。

注意tok/s は 理論値 Roofline であり、製品ベンチマークではない。
C-3

1M Contextは、1Mの便利な記憶ではない

1M Context対応は、一度に大量の情報を渡せる能力であり、安価で永続的な記憶を意味しない。Contextを長くすると、回答前の読み込み時間、KVキャッシュ、1トークン当たりのKV読み出し、入力処理コストが増え、同時に処理できる利用者が減る。情報量が増えるほど、必要な記述を正確に利用できるとは限らない。

📈

Prefill 増

応答が遅くなる

💾

KV 増

同時実行数が減る

🎯

精度低下

必要な記述を使えない

💰

課金 増

入力トークン増

GLM-5.2について、576次元のlatent KVを78層でBF16保持すると仮定すると、1,048,576トークンのMLAコアKVだけで1 Sequence当たり約87.75GiBとなる。これはDSA Indexer、MTP、Runtime Workspace、Allocatorなどを含まない概算である。MLAで圧縮しても、1ユーザーが1M Contextを占有することの容量影響は大きい。

経営判断としては、公称Contextを最大限使うことではなく、Application contextを業務ごとに定めるべきである。長期情報は外部のMemory Serviceへ保存し、Context Builderが今回必要な部分だけを選ぶ。長Contextは必要な案件へ限定して割り当てる資源であり、会話ログを無制限に保存する代替手段ではない。

注意概算 1M Context は MLAコアだけで1 Sequenceあたり約87.75GiB。1M対応 ≠ 1Mを常用。
C-4

“会話を覚える”サービスの責任主体を決める

会話記憶を導入するときは、機能の有無だけでなく、誰が状態を保存し、削除し、監査するかを決める必要がある。素材で整理された2026年6月時点では、OpenAIは応答やConversationを連結するサーバー側の仕組みを提供し、GoogleもInteractionをサーバー側で継続できる。一方、AnthropicのMessages APIは基本的にステートレスで、Memory Toolの保存処理も開発者側が担う。

提供者会話状態長期状態状態管理主体
OpenAIprevious_response_idConversations APIOpenAI または自前
Googleprevious_interaction_idInteraction objectGoogle
AnthropicMessages配列を毎回送信Memory Tool原則として開発者

方式が異なっても、基本的な流れは「会話を保存する、必要な情報を取得する、現在のContextへ入れる、同じモデル重みで推論する」である。会話が継続して見えることは、モデルが利用者ごとに継続学習していることを意味しない。

APIサービスでもSelf-hostでも、保持期間、データ所在、利用者による削除、案件間の分離、監査ログ、機密区分を明確にする必要がある。ローカルAIを採用する場合は、この責任が自社へ移る。モデルのデプロイだけでなく、Memory Control Planeを運用する体制まで含めて採用可否を判断すべきである。

C-5

Reasoningは品質向上策であり、同時に容量消費策

Reasoningは複雑な問題の成功率を高める可能性がある一方、生成Token、待ち時間、コンテキスト、KVキャッシュを増やす。したがって、Reasoning ONを単純な品質スイッチとして扱うのではなく、追加資源を使って成功率をどこまで改善できるかというトレードオフとして評価する必要がある。

素材に収録された公式文書の単一例では、OpenAIの例はReasoning 1,024トークンとVisible Output約162トークンで、Reasoningなし相当の約7.3倍の生成量になった。Geminiの例ではThought 297トークン、Visible Output 171トークンで約2.74倍であった。これらは平均値ではなく、Reasoningが短い最終回答の背後で大きな生成量になり得ることを示す個別例である。

公式単一例 単一例ReasoningVisible倍率
OpenAI1,024約162約7.3×
Gemini297171約2.74×

AresのTAU-Bench Retailでは、High固定は成功率54.8%で評価全体1,007K Token、動的選択は成功率58.5%で476K Tokenであった。この特定ベンチマークでは、常時Highより動的選択の方が少ないTokenで良い結果を得た。意思決定では「Reasoningを使うか」だけでなく、「どのタスク、どのStep、どの失敗条件でEffortを上げるか」を設計する必要がある。

Effort 特定ベンチ成功率Token消費
常時Low35.0%25K
常時Medium47.3%137K
常時High54.8%1,007K
Ares動的選択58.5%476K

C-6

すべての問い合わせをHighで考えさせない

Reasoningの運用方針では、タスクの難しさと誤答コストに応じてEffortを割り当てる。分類、検索、定型回答はOFFまたはMinimal、通常チャットや要約はLow、コードや表計算、資料作成はMedium、複雑なデバッグや設計判断はHighを初期値にできる。Effortは固定トークン数ではないため、各経路の実測分布を監視する。

一度目から強いモデルとHigh Effortを使うのではなく、Fast ModelとLowで処理し、Verifierが失敗を検出したときにMediumへ上げる。再度失敗した場合にStrong ModelとHighを使い、それでも不十分なときだけ専門Workerを追加する。簡単な仕事が高価な経路を通り続けることを防ぎながら、難しい仕事には必要な資源を割り当てられる。

タスクReasoning
分類・検索OFF / Minimal
通常チャットLow
コード・資料Medium
複雑設計High
長時間AgentHigh / xhigh
 
routes:
  classification:   { model_tier: fast,     reasoning_effort: minimal }
  normal_chat:      { model_tier: standard, reasoning_effort: low }
  code_and_documents:{ model_tier: standard, reasoning_effort: medium }
  complex_design:   { model_tier: strong,   reasoning_effort: high }

escalation:
  on_verifier_failure:
    - raise_effort_one_level
    - use_stronger_model
    - add_specialized_workers

次世代のRouterが決めるのはモデル名だけではない。「どのモデルを使うか」「どのReasoning Effortを使うか」「何Workerを呼ぶか」の組み合わせである。この三つを独立した資源配分変数として管理することで、品質、速度、Token消費、同時実行能力を調整できる。

コール次世代Router ＝ どのモデル × どのEffort × 何Worker。
C-7

ローカルAI採用判断は、モデルではなく運用境界で決める

ローカルAIの採用判断は、「手元のPCでモデルが動くか」だけで行うべきではない。ここでいうローカルAIには、自社のAWSやGCPアカウント内でSelf-hostする構成も含まれる。重要なのは、モデル、会話状態、ログ、文書、監査情報のうち、どこまでを自社の運用境界へ置くかである。

API中心の構成はモデル運用の負担を外部へ移せる一方、保持条件、データ所在、利用可能なモデル、サービス側の制約を確認する必要がある。Self-hostはモデルやデータ経路を細かく制御できるが、容量計画、推論Runtime、障害対応、更新、セキュリティ、監視を自社で担う。どちらが常に優れているという関係ではない。

論点API中心Self-host中心確認事項
データ所在ベンダー側自社境界内機密区分・規制
状態の所有者一部ベンダー自社削除要求対応
モデル選択提供メニュー自由（運用責任も）更新頻度
運用負荷低高人員・スキル
容量計画ベンダー任せ自社で試算ピーク生存トークン
監査ログ依存自社設計追跡可能性

また、モデルをSelf-hostしても会話記憶は自動的に完成しない。Raw Event Store、Session、Structured Memory、Semantic Index、Artifact Store、Context Builder、Memory Writerが別途必要になる。採用判断ではモデル費用だけでなく、このMemory Serviceを継続運用できる人員と統制を含めるべきである。

C-8

マルチエージェントでは、Worker数が新しいコスト倍率になる

マルチエージェントでは、Workerを増やすほど多角的な検討ができる一方、各Workerの入力、Reasoning、出力、KVキャッシュが追加される。全Workerへ同じ長大な履歴を渡す設計では、ContextとKVがほぼWorker数に応じて増え、モデルやEffortの最適化だけでは吸収できない。

望ましい設計では、CoordinatorがShared Task StateとDecision Logを管理し、Plannerには計画に必要な情報、Coderには対象コードと設計、Verifierには要件と成果物だけを渡す。各Workerの成果は、結論、根拠、未解決事項、Artifactへの参照へ圧縮して中央へ戻す。詳細な文書やコードはArtifact Storeに置き、会話へ複製しない。

設計Context量KV影響監査性
全履歴を全WorkerへWorker数倍Worker数倍低（重複ログ）
Blackboard＋Artifact役割分の最小限抑制高（構造化）

意思決定時には、Worker数を品質向上の無料な手段とみなさず、新しいコスト倍率として扱う必要がある。Workerごとの平均Context、Reasoning p95、成功率への寄与を測定し、役割が重複するWorkerは削減する。共有メモリをBlackboard化することは、容量削減だけでなく、判断過程の監査性にもつながる。

C-9

経営ダッシュボードで見るべき指標

Reasoning型サービスは、平均応答時間とoutput_tokensだけでは状態を把握できない。Token量はinput、reasoning、visible output、toolに分け、利用者が見た回答と基盤が実際に処理した量を区別する。速度も、生成開始までのTTFT_any、最初のVisible AnswerまでのTTFV、Reasoningを含むdecode_tok/s_all、画面表示部分のvisible tok/sを分けて見る。

指標何を示すか
input / reasoning / visible_output tokensコスト内訳（見えない生成を分離）
TTFT_any / TTFV生成開始 / 最初の回答までの体感
decode_tok/s_all / visible_tok/s基盤性能 / 体感性能
peak_KV_bytes / 同時Sequence数同時実行の上限要因
requests/sec / task_success_rateスループット / 品質

容量面ではpeak KV bytes、同時Sequence数、Queue長、requests/secを監視する。品質面ではtask success rate、Verifier通過率、再試行率、Effort Escalation率を追う。High Effortによって成功率が上がっても、TTFVやQueueがSLAを超えるなら、サービス全体として最適とは言えない。

ダッシュボードの目的は、一つの性能値を競うことではなく、品質、速度、容量、Token消費の関係を可視化することである。「Visible tokens ÷ 総応答時間」だけで推論性能を比較すると、非表示Reasoningが多いサービスほど見かけ上のtok/sが低くなる。投資判断には、総処理量とユーザー体感の両方が必要である。

C-10

1枚で決めるAIワークロード配置表

AIワークロードの配置は、タスク難度とレイテンシ許容度の二軸で整理できる。難度が低く即時性が高い分類や検索は、Fast ModelとMinimal Reasoningへ配置する。通常チャットや要約はStandard ModelとLow、コードや資料作成はMedium、複雑な設計や調査はStrong ModelとHighの候補になる。

難度 低 × 速さ要
Fast model / Low・Minimal Effort / 同期
難度 高 × 速さ要
Strong model / Medium / Prompt Cache
難度 低 × 遅延許容
バッチ処理 / 低コストTier
難度 高 × 遅延許容
High/xhigh / 複数Worker / 非同期

過去の情報が必要な仕事には、モデルを大きくする前にMemory Retrievalを加える。複数の専門領域をまたぐ仕事には、必要な役割だけWorkerを追加する。品質不足が起きた場合も、最初からすべてを最大構成へ移すのではなく、Effort、モデル、Workerの順序を定めて段階的にEscalationする。

配置表には、モデル名だけでなく、最大Application context、Reasoning方針、期待TTFV、同時実行数、保存するMemory、監査要件を記載する。この一枚を基盤担当、プロダクト担当、ガバナンス担当が共有することで、技術構成とサービス要件を同じ判断表で議論できる。

C-11

技術付録：判断の根拠となる数式と全数値

技術付録は、経営判断へ用いた数値の再現性を確保するために置く。重み容量は「Parameters × bits ÷ 8」、MHA／GQAのKVは「2 × Layers × KV Heads × Head Dimension × dtype bytes × Sequence length」、同時Sequence数は「KVへ使えるVRAM ÷ 1 SequenceのKV」で概算する。Decode Rooflineは、短いContextでは「帯域 ÷ 重み容量」、長いContextでは「帯域 ÷（重み読み出し + KV読み出し）」として扱う。

Reasoning込みの完了時間は、「Queue + Prefill +（Reasoning + Visible Output）÷ Decode tok/s + Tool時間」で概算する。これらの式は設計初期の比較には有効だが、Runtime、Kernel、通信、Batch Schedulerを含む実測を置き換えるものではない。付録の各表には、理論値、仮定に基づく概算、公式文書の単一例、特定ベンチマークの結果という種別を明記する。

主要な数式 
# 重み容量
Weight GB ≒ Params[B] × bits ÷ 8
# KV
KV bytes/token = 2 × L × Hkv × Dh × dtype_bytes
# 同時数
最大Sequence ≒ (VRAM − Weight − Reserve) ÷ (KV/token × 平均長)
# tok/s Roofline
tok/s上限 ≈ 帯域 ÷ (Weight + KV読取)
# Reasoning 時間
Completion ≈ Queue + Prefill + (Reasoning + Visible) ÷ tok/s + Tool
